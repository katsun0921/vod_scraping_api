# VOD情報源 候補調査（X・公式サイト・TMDb）

作成: 2026-07-22
対象: `docs/feature/vod-release-calendar-spec.md` 7.（情報源の管理方針）

## 調査方法・限界

このセッションでも外部サイトへの直接アクセス（WebFetch）は403でブロックされるため
（[theater-sources-candidates.md](theater-sources-candidates.md) と同じ環境制限）、
以下は**Web検索のスニペットのみ**に基づく一次調査であり、規約の正確な全文・条番号は未検証。
**「VOD情報源」シートに登録する前に、必ず人間がブラウザで規約原文を確認すること。**

Katsumascore は Google AdSense で収益化しているため、「商用・有償目的の利用を禁止する規約」
「自動化手段によるアクセスを禁止する規約」の両方に抵触しないことを採用条件とする
（TMDb・PR TIMES撤回時と同じ判断基準）。

## A. 採用（必須）: VODサービス公式Xアカウント

**判断: 採用。X API v2（有償・公式API）経由で取得するため、Webスクレイピングの規約問題が
発生しない唯一の機械取得ルート。**

- 取得手段は既存の `news_bot/fetch_x.py`（X API v2 Bearer Token、`since_id`キャッシュで
  新着分のみ課金対象）を流用する。x-news-bot仕様書4.1のレイヤー4
  「配信サービス（Netflix、Disney+、Prime Video、Hulu、U-NEXT）」として当初から想定されていた枠
- X APIの利用は開発者規約の範囲内での利用であり、スクレイピング禁止条項とは別の話
  （Xの規約がスクレイピングを禁止しているのはWebページの機械取得。APIは公認ルート）
- コスト: Pay-Per-Use従量課金。news-bot-x と同じく1日1回（または週1回）実行 + `since_id`
  キャッシュで新着分のみに抑える。前払い制のため想定外請求はない（x-news-bot仕様書4.6）

### 登録候補アカウント（ハンドルは登録前に人間が実在確認すること）

| サービス | 想定アカウント | 備考 |
|---|---|---|
| Netflix | Netflix Japan 公式 | 配信開始日付きの告知ポストが多い |
| Prime Video | プライムビデオ日本公式 | 月間ラインナップ告知あり |

| Hulu | Hulu Japan公式 | 配信開始告知あり |
| DMM TV | DMM TV公式 | 配信開始告知あり |

X公式アカウントのポストは非構造化テキストのため、取得後にAI抽出
（タイトル・サービス・配信開始日・配信種別の構造化）を挟んでから
「VOD配信予定」シートの候補行にする（仕様書7.3参照）。

### A.1 コスト試算: 1日1回監視（2026-07-22）

**前提（X API Pay-Per-Use、2026年4月20日改定後の料金）**

| 項目 | 単価 | 備考 |
|---|---|---|
| Post read（他者アカウントの投稿読み取り） | $0.005/件 | VOD公式アカウントは自社の投稿ではない（＝Bearer Tokenの所有者自身の投稿ではない）ため「Owned Reads」（$0.001/件）ではなく通常のPost read料金が適用される |
| User read（ハンドル→user_id解決） | $0.010/件 | `fetch_x.py`は`user_id`をシートにキャッシュするため**初回のみ**発生し、2回目以降は発生しない |
| 投稿（Post作成） | $0.015/件（URL付き$0.20/件） | 本施策もx-news-botと同じく**投稿は人間が手動で行う**ため、この課金は発生しない |
| 月間上限 | 200万reads/月 | 本施策の規模では到達し得ない |

Post read課金は「実際にレスポンスで返ってきた投稿件数」に対して発生する
（`since_id`で新着なしなら0件・0円）。`max_results=10`はあくまで1回あたりの上限で、
課金対象件数そのものではない。

**試算（対象6アカウント、1日1回、`since_id`キャッシュ運用）**

| シナリオ | 1アカウントあたりの新着/日 | 6アカウント合計/日 | 日額 | 月額（30日換算） |
|---|---|---|---|---|
| 少ない週（通常運用） | 0.5件 | 3件 | $0.015 | 約$0.45 |
| 平均的な週 | 2件 | 12件 | $0.06 | 約$1.80 |
| 多い週（月初・クール開始等の告知ラッシュ） | 5件 | 30件 | $0.15 | 約$4.50 |

+ 初回のみ user read 6件 × $0.010 = **$0.06**（一度きり）

**結論**: 月額 概ね **$0.5〜$5程度**。既存news-bot-x（公式X一覧6件・地域ごとに1日1回）と
同規模の追加であり、200万reads/月の上限にも遠く及ばない。x-news-bot仕様書4.6の残高アラート
（$1未満でSlack通知）の運用にそのまま乗せられる規模で、Xを必須情報源に採用しても
コスト面のハードルにはならない。

**未確定要素**: 実際の投稿頻度はアカウントごとに異なり運用開始前は推定値でしかない
（上記は仮定ベース）。本番投入後、数週間分の実績（`response.usage`相当のログ、または
`since_id`更新時に取得件数をログ出力する形で計測）を見て試算を更新するのが望ましい
（8.「VOD情報源」シートの取得頻度・`vod_publish`実行頻度の判断材料にもなる）。

## B. 不採用: 各VODサービス公式サイトのスクレイピング

主要サービスの利用規約を調査した結果、**規約文言が確認できた4サービスはいずれも
ロボット・スパイダー・スクレイパー等の自動化手段によるアクセス／データ収集を明示的に禁止**
していた。theater施策で配給会社公式サイトを不採用にしたのと同じ判断基準により、
明示禁止のサービスは情報源に採用しない。

| サービス | 調査結果 | 判断 |
|---|---|---|
| Netflix | 利用規約4.6「ロボット、スパイダー、スクレイパーその他の自動化手段を使用せず」「いかなるデータマイニング、データ収集もしくは抽出方法をも利用しない」と明示 | **不採用** |
| Amazon（Prime Video） | Amazon.co.jp利用規約の利用許可範囲から「データマイニング、ロボットなどのデータ収集・抽出ツールの使用」を明示的に除外。商用目的でのコンテンツ収集・利用も不許可 | **不採用** |
| Disney+ | 禁止行為として「自動化された手段（ロボット、スパイダー、スクレイパー等）を使用したコンテンツへのアクセス・監視・複製」を明示（情報解析目的の利用も規約許諾範囲外と明記） | **不採用** |
| Hulu（日本） | 「ロボット」「スパイダー」「オフラインリーダー」等の自動化手段によるアクセスを禁止（一般に入手可能なサーチインデックス作成目的の限定例外はあるが、本施策は該当しない） | **不採用** |
| U-NEXT | 規約全文をスニペットで確認できず（複製・不正アクセス禁止の一般条項は示唆される）。**人間がブラウザで規約原文を確認するまで保留** | 保留 |
| DMM TV | DMM本体の会員規約は未確認（DMM GAMES等の個別規約ではスクレイピング・クローリングを明示禁止しており、本体も同様の可能性が高い）。**人間が確認するまで保留** | 保留 |

### `coming-soon-pipeline.md`（`scrape_official.py`）との関係

[coming-soon-pipeline.md](coming-soon-pipeline.md)の`scrape_official.py`は
Netflix（`https://www.netflix.com/jp/coming-soon`）・U-NEXT
（`https://video.unext.jp/browse/new-arrivals`）・Prime Video
（`https://www.amazon.co.jp/gp/video/offers`）の3サービスを対象に実装済み（未使用）だが、
上表の判断と照らすと：

- **Netflix・Prime Video分は組み込めない**（規約で自動化手段によるアクセスを明示的に禁止）
- **U-NEXT分は保留中の規約確認が完了すれば転用できる**。`requests + BeautifulSoup4`の
  実装がそのまま使えるため、確認が取れ次第「VOD情報源」シートに`取得方式=html`で
  登録し、`scrape_official.py`の該当関数を`news_bot/`側に移植すればよい

TMDb discover API（同ファイルの`_fetch_tmdb()`相当は`tmdb_upcoming.py`）は
C.節の判断どおりTMDb商用契約が未成立のため使わない。

### 既存 `vod_bot` との関係についての注記

`vod_bot`（配信状況チェッカー）は既に各サービスの作品ページへ自動アクセスしているが、
これは既存投稿の配信状況確認という既運用であり、本施策の判断とは切り離して扱う。
**本施策（新規コンテンツ収集・サイト掲載目的）では theater 施策で確立した保守的基準を適用し、
明示禁止のサービスを情報源に加えない。** 既存 vod_bot の運用リスクの再評価は本施策のスコープ外
（必要なら別途人間が判断する）。

## C. 不採用: TMDb

**判断: スクレイピング・API とも現状は不採用。**

1. **Webサイトのスクレイピング**: TMDb利用規約はサイトからのスクレイピングを明示的に禁止
   （コンテンツの copy / scrape / distribute 等を個人的・非商用目的の明示的許可範囲外で禁止）。
   公式に認められたプログラムアクセス手段は**APIのみ**
2. **API**: [theater-sources-candidates.md](theater-sources-candidates.md) A.節で確定済みのとおり、
   AdSense収益化サイトでの利用は「Personal Use（非商用）」の誓約に反するため無償利用不可。
   Commercial APIプラン（年商$1M未満で$149/月）を契約した場合のみ利用可

つまり「APIの商用契約が高いのでスクレイピングで代替する」という選択肢は規約上存在しない。
商用契約が成立した場合は、スクレイピングではなく**API**（coming-soon-pipeline.md の
`/discover` + watch providers 実装）を取得レイヤーとして合流させる。
