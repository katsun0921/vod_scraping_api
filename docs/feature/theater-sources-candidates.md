# 劇場情報源 候補リスト（日本限定・初期調査）

作成: 2026-07-17
対象: `docs/feature/theater-release-calendar-spec.md` レイヤー1データソース（未確定事項#1）

## 調査方法・限界

このセッションのネットワークポリシーにより外部サイトへの直接アクセスが全面ブロックされている
（`WebFetch`・`curl`ともに主要ドメイン宛の接続が403で拒否される。`example.com`宛でも同様のため
サイト固有のブロックではなく環境側の制限）。そのため以下は**Web検索のスニペットのみ**に基づく
一次調査であり、以下は未検証。

- RSSフィードが実際に稼働しているか
- フィードの中身が「公開日付き作品一覧」として構造化されているか、単なるサイト更新通知か
- 利用規約の正確な文言・スクレイピング可否

**「劇場情報源」シートに登録する前に、必ず人間がブラウザ・RSSリーダーで実際に開いて内容と
利用規約を確認すること**（仕様書7.「利用条件：利用規約を確認する」に対応）。

## A. 採用決定・実装済み: TMDb API

| 候補 | 概要 | 取得方式 | 備考 |
|---|---|---|---|
| **TMDb API** `/discover/movie`（`region=JP` + `with_release_type=2,3`） | 公式APIで劇場公開日（type=2先行公開/3劇場公開）を国別に取得できる | `tmdb`（要`TMDB_API_KEY`、実装済み） | 利用規約が明確で、既存 `docs/feature/coming-soon-pipeline.md` で配信系データ用に採用予定のTMDb基盤を流用できる。ミニシアター・単館系邦画の網羅性は未検証で、大型配給作品中心になる可能性が高い |

TMDbは仕様書の非ゴール「全配給会社・全ミニシアター作品の完全網羅を保証しない」と初期MVPの
方針に合致しやすく、法的リスクが最も低い候補として **2026-07-17 に採用を決定し、
`news_bot/fetch_theater.py` に実装済み**。原題・ジャンルもレスポンスから取得できるため、
レイヤー3補完の一部を先取りしている。

**採用判断の記録**: TMDb APIの商用利用（広告収益等でのマネタイズ）に該当するかの判定基準は
公式ドキュメント上曖昧で、TMDBサポート（`api@themoviedb.org`）への問い合わせ回答は本決定時点
で未受領。[Redditの実例](https://www.reddit.com/r/iOSProgramming/comments/1jpke10/tmdb_now_has_a_commercial_plan/)
（トラフィックが小さい間は無償利用が黙認されているという報告）を根拠に、公式回答を待たず
無償利用前提で進める判断とした。これはTMDB公式の保証ではなくコミュニティの実例に基づく判断
であることに留意し、トラフィック増加時・公式回答受領時は商用ライセンスの要否を再検討すること。

### TMDb APIの費用（要最終確認）

| 利用形態 | 費用 |
|---|---|
| 非商用利用 | 無料。TMDB出典表示（attribution）義務あり |
| 商用利用（広告収益等でマネタイズしている場合） | 別途ライセンス契約が必要。年間売上$1M未満は $149/月（Commercial APIプラン）、$1M超は個別見積もり |
| レート制限 | 従量課金ではなくIPベースで秒間約40リクエストの上限のみ |

**注意**: 「広告表示によるサイトへの集客・収益化」を商用利用とみなす、という公式記述がある一方、
TMDB担当者のフォーラム回答では「広告自体は問題ない」という揺れた発言も見られ、線引きが曖昧。
Katsumascoreが何らかの形でマネタイズしている場合は商用利用に該当する可能性があるため、
`api@themoviedb.org` へ問い合わせて確認することを推奨する。この判定は
`docs/feature/coming-soon-pipeline.md`（同じ`TMDB_API_KEY`を使う想定）にも影響する。

### 唯一見つかった構造化ルート: PR TIMES企業別RSS

配給会社が自ら配信するプレスリリースサービス「PR TIMES」には企業別RSSが存在する
（例: 東宝 `company_id=27367` → `https://prtimes.jp/companyrdf.php?company_id=27367`）。
PR TIMESは配信・シンジケーションを前提としたサービスのため直接スクレイピングよりは
法的リスクが低いと考えられるが、以下の制約がある。

- 中身は「公開作品一覧」ではなく個別プレスリリース（新作発表・興行成績・タイアップ告知等が
  混在）で、`fetch_theater.extract_release_date()`と同様のテキスト解析が必要
- 東宝のPR TIMES利用は確認できたが、東映・松竹・ギャガ等が同様に利用しているかは未確認
  （配給会社ごとに個別調査・company_id特定が必要）

レイヤー2の個別作品情報源としては有望だが、レイヤー1の代替にはならない。
