# Katsumascore Xニュース自動投稿システム 実装仕様書（Ver.1.4）

対象読者：Claude Code（実装エージェント）
前バージョン：Ver.1.0（設計方針）・Ver.1.1/Ver.1.2/Ver.1.3からの変更点は末尾「変更履歴」参照

---

## 1. 目的

Katsumascore（映画・アニメ・ドラマレビューメディア）のX運用効率化のため、映画・アニメ関連ニュースを自動収集し、AI判定を経てXへ投稿するシステムを構築する。

Googleスプレッドシートを簡易データベースとして利用し、将来的にPostgreSQLへ移行可能な設計とする。

---

## 2. システム全体像

```
[ニュース取得: RSS優先/一部スクレイピング]
        │
        ▼
[重複チェック（URL + タイトル類似度）]
        │
        ▼
[Googleスプレッドシート保存]
        │
        ▼
[AI判定: Claude API（S/A/B/D）]
        │
    ┌───┴───┐
    S・A          B・D
    │             │
    ▼             ▼
[投稿文生成]    [保存のみ]
    │
    ▼
[人間承認ステップ（後述4.4）]
    │
    ▼
[X投稿（本文＋リプライURL方式）]
    │
    ▼
[投稿履歴シートに記録]
```

---

## 3. 技術スタック

| レイヤー | 技術 | 備考 |
|---|---|---|
| ニュース取得 | Python: `feedparser`（RSS） / `requests` + `BeautifulSoup`（HTML） | RSS提供元を優先。スクレイピングは規約確認済みの媒体のみ |
| データストア | Google Sheets API（`gspread`） | MVP。将来PostgreSQL移行前提でテーブル設計する |
| AI判定・文生成 | Claude API | 判定はfew-shotで安定化（4.3参照） |
| X投稿 | X API v2（Pay-Per-Use従量課金） | 本文＋リプライ分割でコスト最適化（4.5参照） |
| 実行トリガー | GitHub Actions（cron、1〜2時間おき推奨） | セルフホスト/n8nでも可 |
| 通知 | Slack Webhook | 承認依頼・エラー通知・残高アラート |

**注意：この開発環境（Claude.aiのコード実行）はネットワークが許可ドメインのみに制限されており、Google Sheets API・X API・ニュースサイトへの直接アクセスは不可。実際の常駐実行はGitHub Actions等の外部環境で行う。**

## 4. 実装詳細（レビュー反映）

### 4.1 ニュース取得

- 取得方法は「RSS > 公式API > HTMLスクレイピング」の優先順位とする
- 対象媒体ごとにRSS提供の有無・利用規約を確認してから`ニュースソース`シートに登録する
- レイヤー構成：
  - レイヤー1（最優先）：作品公式X、制作会社、配給会社、公式サイト
  - レイヤー2：映画.com、シネマトゥデイ、コミックナタリー、アニメ！アニメ！、Variety、Deadline
  - レイヤー3：公開予定作品・放送中アニメの個別監視（公式X・ハッシュタグ）
  - レイヤー4：配信サービス（Netflix、Disney+、Prime Video、Hulu、U-NEXT）

### 4.2 重複チェック

- 一次チェック：URL完全一致
- 二次チェック：タイトルの類似度判定（同一ニュースを複数媒体が報じるケース対応）
  - 実装案：Claude APIに「この記事は直近24時間の既存記事と同一ニュースか」を判定させる、またはタイトルの embedding 類似度で閾値判定
- 重複と判定された記事は`ニュース取得`シートに`重複`フラグを立てて保存のみ（投稿対象外）

### 4.3 AI判定基準（S/A/B/D）

Claude APIへのプロンプトには過去の判定実例（few-shot）を含め、判定のブレを抑える。

| ランク | 内容 | 扱い |
|---|---|---|
| S | 新作発表・続編決定・公開日決定・予告公開・キャスト発表・配信開始 | 即投稿対象 |
| A | インタビュー・興行収入・制作情報 | 投稿対象 |
| B | グッズ・キャンペーン・小規模ニュース | 保存のみ |
| D | 未確認情報・噂・重複記事 | 除外 |

判定プロンプトには以下を含めること：
- 判定基準の説明
- 過去のS/A/B/D判定の具体例（各2〜3件）
- 出力フォーマット指定（JSON: `{rank, reason, confidence}`）

**AI判定の精度比較テスト（新規・フェーズ1範囲外の検証用）**：Claude単独判定に加え、ChatGPT・Grokでも同一プロンプトで並列判定させ、結果を「ニュース取得」シートに並べて記録できる（`news_bot/judge.py` + `news_bot/ai_clients.py`、`NEWS_BOT_JUDGE_PROVIDERS`環境変数で有効化）。最終的にどの判定を投稿判断に採用するかは`NEWS_BOT_JUDGE_DECISION`で制御し、既定では引き続きClaude単独判定（`primary`）。比較結果を見た上で多数決方式（`majority`）への移行を検討する。

### 4.4 人間承認ステップ（新規追加・重要）

Ver.1.0では S・A 判定後に投稿文生成→X投稿が完全自動だったが、誤情報・未確認ニュースの誤投稿リスクを避けるため以下を追加する。

- **S判定・A判定とも**：Slack通知し、承認リアクション（:white_check_mark:）があってから投稿。自動投稿は行わない
- 取り消しは :x: リアクション。未反応の場合は保留のまま次回cronで再確認する
- 将来的に誤判定率が低いと確認できた場合、S判定のみ自動投稿の再導入を検討

### 4.5 X投稿のコスト最適化

2026年のX API料金改定でURL付き投稿は$0.20/件と高額（通常投稿は$0.01/件）。対策として：

- メイン投稿：ニュース本文のみ（URLなし）
- リプライ：URLを含める（$0.01/件のまま据え置き）
- `X投稿履歴`シートは「本文」「リプライ本文（URL含む）」を別カラムで管理する設計に変更

### 4.6 コスト管理

- X Developer Consoleで支出上限を設定
- 残高が一定額（例：$1）を下回ったらSlack通知
- Pay-Per-Useは前払い制で自動課金されないため、残高切れ＝投稿停止のみ（想定外請求なし）

### 4.7 Coming Soonソース連携（フェーズ2以降・新規）

`vod_scraping_api`リポジトリの別パイプライン「Coming Soon パイプライン」（[docs/coming-soon-pipeline.md](./coming-soon-pipeline.md)、`coming-soon/`）が出力する`StreamingEvent`JSON（Cloudflare R2 or Cloud Storage）を、RSSと並ぶ第2のニュースソースとして取り込む。

- **対象イベント**：`status`が`scheduled`→`available`に遷移した回のみ（＝配信開始）を検知する
- **ランクはS固定**：仕様書4.3のS判定基準に「配信開始」が明示されているため、AI判定（judge.py）をスキップしS確定とする。Claude API呼び出しが不要になりコスト削減にもなる
- **重複チェック**：外部ニュースと異なりURLを持たない場合があるため、URL完全一致ではなく`event_id`をキーにする
- **投稿文生成**：`title_ja` / `overview_ja` / `genres` / `duration_min`等の構造化メタデータをcompose.pyに渡し、RSS由来より情報量の多い投稿文を生成する
- **リプライURL**：外部媒体のURLではなく、`wp_post_id`から解決したkatsumascore自身のレビュー記事URLを使用する。ニュースの転載ではなく自サイトへの送客を主目的とする、news_bot本来の狙いに最も合致する使い方
- **ニュースソースシートの拡張**：「取得方法」列に`RSS`/`HTML`/`API`に加え`JSON`を追加し、Coming Soonパイプラインの出力先URLを登録する

**前提条件**：Coming Soonパイプライン側の未決定事項4件（ストレージ選択・`tmdb_id`メタの有無・U-NEXT TMDbプロバイダーID・WP Application Password発行）の解決、および実装完了が前提。それまではフェーズ2以降の拡張として扱い、フェーズ1（MVP）のスコープには含めない。

## 5. Googleスプレッドシート構成

### ① タイトル一覧（管理対象作品）
`ID / タイトル / 種類（映画・TVアニメ・配信など） / 状態（公開予定・公開中・放送中・配信中・終了） / 公開日 / 優先度`

### ② 公式X一覧
`タイトルID / アカウント名 / URL / 種別（作品・配給・制作会社・配信サービス）`

### ③ ニュースソース
`ID / 名称 / URL / カテゴリ / 取得方法（RSS・HTML・API・JSON（新規：4.7 Coming Soonソース）） / 取得間隔 / 有効/無効 / 規約確認済み（新規）`

### ④ ニュース取得
`取得日時 / タイトル / URL / 媒体 / 関連タイトル / 概要 / 重要度(S/A/B/D) / 投稿状態 / 重複フラグ（新規）/ AI判定理由（新規）`

> Coming Soonソース由来の行はURLを持たない場合があるため、URL列は空欄可とし`event_id`で識別する（4.7参照）。

### ⑤ X投稿履歴
`投稿日時 / ニュースID / 本文（新規：URLなし） / リプライ本文（新規：URL含む） / インプレッション（将来） / いいね数（将来） / 承認者（新規）`

### ⑥ YouTube Shorts
`動画タイトル / URL / 公開日 / X投稿済み`
