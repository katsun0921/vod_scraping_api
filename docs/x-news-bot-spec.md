# Katsumascore Xニュース自動投稿システム 実装仕様書（Ver.1.19）

対象読者：Claude Code（実装エージェント）
前バージョン：Ver.1.0（設計方針）・Ver.1.1/Ver.1.2/Ver.1.3/Ver.1.4/Ver.1.5/Ver.1.6/Ver.1.7/Ver.1.8/Ver.1.9/Ver.1.10/Ver.1.11/Ver.1.12/Ver.1.13/Ver.1.14/Ver.1.15/Ver.1.16/Ver.1.17/Ver.1.18からの変更点は末尾「変更履歴」参照

---

## 1. 目的

Katsumascore（映画・アニメ・ドラマレビューメディア）のX運用効率化のため、映画・アニメ関連ニュースを自動収集し、AI判定を経てXへ投稿するシステムを構築する。

Googleスプレッドシートを簡易データベースとして利用し、将来的にPostgreSQLへ移行可能な設計とする。

---

## 2. システム全体像

```
[ニュース取得: RSS優先/一部スクレイピング]
        │
        ▼
[重複チェック（URL + タイトル類似度）]
        │
        ▼
[Googleスプレッドシート保存]
        │
        ▼
[AI判定: Claude API（S/A/B/D）]
        │
    ┌───┴───┐
    S・A          B・D
    │             │
    ▼             ▼
[投稿文生成]    [保存のみ]
    │
    ▼
[人間承認ステップ（後述4.4）]
    │
    ▼
[X投稿（本文＋リプライURL方式）]
    │
    ▼
[投稿履歴シートに記録]
```

---

## 3. 技術スタック

| レイヤー | 技術 | 備考 |
|---|---|---|
| ニュース取得 | Python: `feedparser`（RSS） / `requests` + `BeautifulSoup`（HTML） | RSS提供元を優先。スクレイピングは規約確認済みの媒体のみ |
| データストア | Google Sheets API（`gspread`） | MVP。将来PostgreSQL移行前提でテーブル設計する |
| AI判定・文生成 | Claude API | 判定はfew-shotで安定化（4.3参照） |
| X投稿 | X API v2（Pay-Per-Use従量課金） | 本文＋リプライ分割でコスト最適化（4.5参照） |
| 実行トリガー | GitHub Actions（cron、1〜2時間おき推奨） | セルフホスト/n8nでも可。実装（`news-bot.yml`）は運用判断で1日1回に変更（4.1・変更履歴Ver.1.11参照） |
| 通知 | Slack Webhook | 承認依頼・エラー通知・残高アラート |

**注意：この開発環境（Claude.aiのコード実行）はネットワークが許可ドメインのみに制限されており、Google Sheets API・X API・ニュースサイトへの直接アクセスは不可。実際の常駐実行はGitHub Actions等の外部環境で行う。**

## 4. 実装詳細（レビュー反映）

### 4.1 ニュース取得

- 取得方法は「RSS > 公式API > HTMLスクレイピング」の優先順位とする
- 対象媒体ごとにRSS提供の有無・利用規約を確認してから`RSS一覧`シートに登録する
- レイヤー構成：
  - レイヤー1（最優先）：作品公式X、制作会社、配給会社、公式サイト
  - レイヤー2：映画.com、シネマトゥデイ、コミックナタリー、アニメ！アニメ！、Variety、Deadline
  - レイヤー3：公開予定作品・放送中アニメの個別監視（公式X・ハッシュタグ）
  - レイヤー4：配信サービス（Netflix、Disney+、Prime Video、Hulu、U-NEXT）

**公式Xアカウントの投稿取得（Ver.1.6で本実装）**：レイヤー1の「作品公式X」はRSSを提供しないため、`news_bot/fetch_x.py`でX API v2（Bearer Token）から直接取得する。「公式X一覧」シートに登録されたアカウントのうち、地域列（`日本`/`アメリカ`）が一致し有効なものを`main.py`の`fetch_x_cycle(region)`が巡回し、以降は`fetch_cycle()`と同じ重複チェック→AI判定→Slack通知の処理に合流する。コスト管理のため地域ごとに1日1回の実行を想定し、`since_id`（前回取得した最新投稿ID）をシートにキャッシュして新着分のみ課金対象にする。

### 4.2 重複チェック

- 一次チェック：URL完全一致
- 二次チェック：タイトルの類似度判定（同一ニュースを複数媒体が報じるケース対応）
  - 実装案：Claude APIに「この記事は直近24時間の既存記事と同一ニュースか」を判定させる、またはタイトルの embedding 類似度で閾値判定
- 重複と判定された記事は`ニュース取得`シートに`重複`フラグを立てて保存のみ（投稿対象外）

### 4.3 AI判定基準（S/A/B/D）

Claude APIへのプロンプトには過去の判定実例（few-shot）を含め、判定のブレを抑える。

| ランク | 内容 | 扱い |
|---|---|---|
| S | 新作発表・続編決定・公開日決定・予告公開・キャスト発表・配信開始 | 即投稿対象 |
| A | インタビュー・興行収入・制作情報 | 投稿対象 |
| B | グッズ・キャンペーン・小規模ニュース | 保存のみ |
| D | 未確認情報・噂・重複記事 | 除外 |

判定プロンプトには以下を含めること：
- 判定基準の説明
- 過去のS/A/B/D判定の具体例（各2〜3件）
- 出力フォーマット指定（JSON: `{rank, reason, confidence}`）

**バッチ判定（Ver.1.17）**：記事1件ごとに1リクエストを叩くとsystemプロンプトの重複送信でコストが嵩むため、`judge.judge_batch()`が重複チェック後の新規記事を`_BATCH_SIZE`（既定15）件ごとにまとめ、1プロバイダーにつき1リクエストで判定する。入出力とも`{"index": ..., ...}`のJSON配列（`judge_system_prompt.md`）で、応答の件数・indexが一致しない場合や不正なrankの場合はその記事のみD判定にフォールバックする。同じバッチ内の記事同士を比較できるため、複数媒体が同一ニュースを報じるケース（D判定＝重複）の精度向上も期待できる副次効果がある。`main.py`の`_process_entries()`は記事ごとの`judge.judge()`呼び出しから`judge_batch()`一括呼び出しに変更した。

**AI判定の精度比較テスト（新規・フェーズ1範囲外の検証用）**：Claude単独判定に加え、ChatGPTでも同一プロンプトで並列判定させ、結果を「ニュース取得」シートに並べて記録できる（`news_bot/judge.py` + `news_bot/ai_clients.py`、`NEWS_BOT_JUDGE_PROVIDERS`環境変数で有効化）。最終的にどの判定を投稿判断に採用するかは`NEWS_BOT_JUDGE_DECISION`で制御し、既定では引き続きClaude単独判定（`primary`）。比較結果を見た上で多数決方式（`majority`）への移行を検討する。Grokは判定差異の傾向を確認できたため比較対象から外し、Claude・ChatGPTの2社比較に絞った（Ver.1.16）。

**Claude APIのprompt caching**：`ai_clients.call_claude()`（judge判定）と`compose.compose_headline()`（スレッド見出し生成）は、`system`プロンプトに`cache_control: {"type": "ephemeral"}`を付けている。1回のrun内で同じsystemプロンプトを記事件数分繰り返し送るため、2件目以降は入力トークンがキャッシュ読み込み（約1/10のコスト）になる。ヒット状況は`response.usage.cache_read_input_tokens`をログ出力して確認できる（`ai_clients.py`）。systemプロンプトがモデルの最小キャッシュ対象トークン数を下回る場合はキャッシュが効かない点に注意（Ver.1.16）。

### 4.4 投稿ステップ（現在は手動投稿・スレッドまとめ方式、Ver.1.5/Ver.1.12で変更）

Ver.1.0では S・A 判定後に投稿文生成→X投稿が完全自動だったが、誤情報・未確認ニュースの誤投稿リスクを避けるため人間の関与を必須とする。

- **現在の運用（手動投稿・スレッドまとめ）**：1回のrun（`fetch_cycle()`/`fetch_x_cycle()`）でS/A判定になった記事は、記事ごとに個別投稿するのではなく**1つのXスレッド（連投）にまとめて**Slackへ通知する（`approval.notify_manual_thread`）。各記事は`compose.compose_headline(entry, rank)`が生成する短い見出し文＋その記事のURLで1行になり、`compose.pack_thread()`が1ツイート分の文字数上限（既定150字、行の途中では分割しない）ごとに機械的にパッキングして連投パーツに分ける。見出し文は判定ランクに応じてトーンを変える：S（速報性が高いニュース）は速報であることが伝わる書き方、A（速報性が中程度のニュース）は控えめに事実を伝える書き方にする（`thread_headline_system_prompt.md`）。見出し文生成のペルソナはSNSマーケティング担当者とし、固有名詞・具体的な数字を活かしてタイムライン上でタップしたくなる文章を意識させつつ、誇張・扇情的な表現やミスリードは避け事実ベースを保つ制約は維持する（Ver.1.18）。実際のX投稿は人間が手動で、パーツを①→②→③…の順に（2件目以降は直前の投稿への返信として）連投する。ボットによる自動投稿・自動承認判定は行わない
- **記事1件だけの単独投稿版（現在未使用・コードは保持）**：`compose.compose()`（本文＋リプライの2分割生成）と`approval.notify_manual_post()`は記事1件を独立して投稿する場合向けに残してある。現在の`main.py`のフローからは呼ばれていない
- **予算状況次第で再開できる自動投稿フロー（現在は無効化・コードは保持）**：承認リアクション（:white_check_mark:）があってから投稿、:x:で取り消し、未反応は保留のまま次回cronで再確認、という`notify_pending`/`resolve`/`process_pending`/`post_x.post_with_reply`の一連の仕組みは`news_bot/`に残してある。再開する際は`main.py`内のコメントを参照して呼び出しを戻すこと

### 4.5 X投稿のコスト最適化

2026年のX API料金改定でURL付き投稿は$0.20/件と高額（通常投稿は$0.01/件）だが、**投稿は人間が手動で行うためこの従量課金は現状発生しない**（課金が発生するのはX API経由で投稿した場合のみ）。そのためVer.1.12でスレッドまとめ方式に変更した際、各記事のツイートに直接そのURLを含める設計にした（`compose.compose_headline()` + URL）。自動投稿を再開する場合は、このコスト最適化（本文にURLを含めずリプライにまとめる）が再び重要になる：

- メイン投稿：ニュース本文のみ（URLなし）
- リプライ：URLを含める（$0.01/件のまま据え置き）
- `X投稿履歴`シートは「本文」「リプライ本文（URL含む）」を別カラムで管理する設計に変更

### 4.6 コスト管理

- X Developer Consoleで支出上限を設定
- 残高が一定額（例：$1）を下回ったらSlack通知
- Pay-Per-Useは前払い制で自動課金されないため、残高切れ＝投稿停止のみ（想定外請求なし）

### 4.7 Coming Soonソース連携（フェーズ2以降・新規）

`vod_scraping_api`リポジトリの別パイプライン「Coming Soon パイプライン」（[docs/drop/coming-soon-pipeline.md](./drop/coming-soon-pipeline.md)。
**2026-07-22、規約上の理由（TMDb API商用契約未成立・依存先スクレイパーが全対象サービスで規約違反と判明）で
`docs/drop/`へ移動・凍結済み**）が出力する予定の`StreamingEvent`JSON（Cloudflare R2 or Cloud Storage）を、
RSSと並ぶ第2のニュースソースとして取り込む構想だった。

- **対象イベント**：`status`が`scheduled`→`available`に遷移した回のみ（＝配信開始）を検知する
- **ランクはS固定**：仕様書4.3のS判定基準に「配信開始」が明示されているため、AI判定（judge.py）をスキップしS確定とする。Claude API呼び出しが不要になりコスト削減にもなる
- **重複チェック**：外部ニュースと異なりURLを持たない場合があるため、URL完全一致ではなく`event_id`をキーにする
- **投稿文生成**：`title_ja` / `overview_ja` / `genres` / `duration_min`等の構造化メタデータをcompose.pyに渡し、RSS由来より情報量の多い投稿文を生成する
- **リプライURL**：外部媒体のURLではなく、`wp_post_id`から解決したkatsumascore自身のレビュー記事URLを使用する。ニュースの転載ではなく自サイトへの送客を主目的とする、news_bot本来の狙いに最も合致する使い方
- **登録方法の見直しが必要**：Ver.1.8で「RSS一覧」シートはRSS専用（取得方法列を削除）としたため、当初案にあった「取得方法列にJSONを追加してComing Soon出力先URLを登録する」方式は使えなくなった。実装時にはComing Soonソース専用の別シート（またはコード内設定）を用意する

**前提条件（凍結中）**：Coming Soonパイプライン側の未決定事項4件（ストレージ選択・`tmdb_id`メタの有無・
U-NEXT TMDbプロバイダーID・WP Application Password発行）の解決だけでは不十分になった。
パイプラインが依存するTMDb APIが商用契約（$149/月〜）未成立のため無償利用不可と判明し、
代替に想定していた各VODサービス公式サイトのスクレイピングも規約で禁止されていることが判明したため、
**本節の連携構想はTMDb商用契約が成立するまで着手不可**。それまではフェーズ2以降の拡張として扱い、
フェーズ1（MVP）のスコープには含めない。なお`vod_scraping_api`側では本連携とは別に、
TMDb・スクレイピングに依存しないVOD配信情報収集の仕組み
（[docs/feature/vod-release-calendar-spec.md](./feature/vod-release-calendar-spec.md)、
AI Web検索 + X公式アカウント）を並行して進めている。

## 5. Googleスプレッドシート構成

### ① タイトル一覧（管理対象作品）
`ID / タイトル / 種類（映画・TVアニメ・配信など） / 状態（公開予定・公開中・放送中・配信中・終了） / 公開日 / 優先度`

### ② 公式X一覧
`ID / アカウント名 / Xハンドル / URL / 種別（作品・配給・制作会社・配信サービス・メディア（新規）） / 地域（日本・アメリカ） / 有効/無効（チェックボックス） / user_id（新規：キャッシュ） / since_id（新規：キャッシュ） / 最終取得日時（新規）`

> 種別「メディア」は、作品公式ではなく第三者のアニメニュースアグリゲーター（例：AniTrendz）向けの区分。

> `user_id`・`since_id`はXのsnowflake ID（19桁の整数文字列）のため、Sheetsの数値変換による桁落ちを避けてraw文字列として書き込む（`news_bot/sheets.py`の`update_x_account_state`）。

> 「有効/無効」列はSheetsのチェックボックス（真偽値セル）で管理する。Ver.1.9で一度廃止したが、Ver.1.10で復活（4.1）。コード側は`row.get("有効/無効") is True`のみを有効と判定するため、テキストで"有効"等と入力しても対象にならない点に注意。

### ③ RSS一覧（Ver.1.5までは「ニュースソース」という名称だった）
`ID / 名称 / URL / カテゴリ / 地域 / 取得間隔 / 有効/無効（チェックボックス） / 規約確認済み（新規）`

> Ver.1.8で「取得方法」列を削除しRSS専用シートとした（4.7参照。Coming Soonソース等の非RSSソースは別シートで管理する）。「有効/無効」列はVer.1.9で一度廃止したが、Ver.1.10で復活（上記②と同じチェックボックス方式）。「地域」列は運用上の整理用に追加されたもので、コードは参照しない。

### ④ ニュース取得
`取得日時 / タイトル / URL / 媒体 / 関連タイトル / 概要 / 重要度(S/A/B/D) / 投稿状態 / 重複フラグ（新規）/ AI判定理由（新規）`

> Coming Soonソース由来の行はURLを持たない場合があるため、URL列は空欄可とし`event_id`で識別する（4.7参照）。

### ⑤ X投稿履歴
`投稿日時 / ニュースID / 本文（新規：URLなし） / リプライ本文（新規：URL含む） / インプレッション（将来） / いいね数（将来） / 承認者（新規）`

### ⑥ YouTube Shorts
`動画タイトル / URL / 公開日 / X投稿済み`

## 7. 現在登録済みのソース一覧

RSS一覧・公式X一覧の現在の登録内容はGoogleスプレッドシート（`RSS一覧`/`公式X一覧`シート）を正とする。このドキュメントには実データのスナップショットは持たない（Sheets側の変更のたびにドキュメントを追従させるコストを避けるため）。

RSS媒体の利用規約に関する調査結果（Ver.1.13時点）：Deadline（PMC運営）はRSS専用条項（利用規約19条）で明示的にRSS利用を許可（要出典表示・元記事への直接リンク）。シネマトゥデイはRSS個別の許可規定がなく、一般条項の「私的使用超の頒布は事前許諾必要」という保守的な書きぶりだが、公開RSSを提供している実態を踏まえ運用判断で規約確認済みとした。Varietyは規約本文を確認できていない。

「公式X一覧」の`種別`列は仕様書5.②の想定（作品・配給・制作会社・配信サービス・メディア）と異なり、季節タグ（例：`2026年夏アニメ`）として運用されている。コードは`種別`をフィルタに使わないため動作に影響しない。
