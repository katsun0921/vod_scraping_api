# 週次パッチ タイムアウト事象と対応（2026-07）

## 概要

週次パッチ（`weekly_patch.py`、GitHub Actions `Weekly VOD Patch`）が、
GitHub Actions のジョブタイムアウト（`timeout-minutes: 120`）により
**バッチ内の投稿を最後まで処理できずに強制キャンセルされる事象**が
複数回発生していたことが判明した。結果として、当該バッチの一部投稿は
その月に一度もスクレイピングされていなかった。

対応として `BATCH_COUNT` を 4 → 8 に変更し、1バッチあたりの処理件数を
半減させることで、全件カバーの周期を「1ヶ月で1周」から「2ヶ月で1周」に
変更した。

---

## 原因

### 直接原因: GitHub Actions のジョブタイムアウト

`.github/workflows/weekly-patch.yml` の `timeout-minutes: 120` により、
実行開始からちょうど120分でジョブが強制キャンセルされる
（`##[error]The operation was canceled.`）。

実行履歴（直近6回）を確認したところ、**5回中4回がタイムアウトでキャンセル**されていた。

| 実行日 | batch | 対象件数 | 結果 | 実処理時間 | 着手済み※1 | 完了済み※2 |
|---|---|---|---|---|---|---|
| 2026-05-31 | 3 | 118件 | ✅ 成功 | 78分 | 118件 | 108件 |
| 2026-06-07 | — | — | ❌ cancelled | 未詳細調査 | — | — |
| 2026-06-14 | — | — | ❌ cancelled | 未詳細調査 | — | — |
| 2026-06-21 | 2 | 109件 | ❌ cancelled | 120分で打ち切り | 103件 | 101件 |
| 2026-06-28 | — | — | ❌ cancelled | 未詳細調査 | — | — |
| 2026-07-05 | 0 | 134件 | ❌ cancelled | 120分で打ち切り | 104件 | 102件 |

※1 対象件数のうち、投稿ループに入った件数（ログの `badge=` 行の件数）。
※2 Phase 3（クールダウン更新）まで到達した件数（ログの `COOLDOWN` 行の件数）。
「対象件数 - 着手済み」が、そのバッチで一度もチェックされなかった投稿数。

> 2026-06-21・2026-07-05 とも、開始時刻からきっちり約120分後
>（119分41秒〜119分45秒）でキャンセルされており、手動キャンセルではなく
> `timeout-minutes` によるものと断定できる。

### 根本原因: サービス切り替え待機のオーバーヘッド

`utils/rate_limit.py` の `RateLimiter.wait_service_switch()` により、
直前と異なる VOD サービスをチェックする際は **追加で10秒待機**する。

投稿ごとに対象サービスの組み合わせが異なるため、ほぼ全チェックが
「切り替え」として扱われる。2026-07-05 の実行では **366回のチェックに対し
365回（99.7%）が切り替え扱い**で、10秒待機だけで約61分を消費していた。

これに加えて、JustWatch による URL 発見が進むにつれて
1投稿あたりの平均チェックサービス数が増加（missing_services が
url_services に組み込まれるため）しており、2026-05-31 は
118投稿で233回のチェック（約2.0回/投稿）だったのに対し、
2026-07-05 は104投稿（未完了）で366回のチェック（約3.5回/投稿）と、
1投稿あたりの負荷そのものも増えていた。

これらが重なり、投稿数・バッチサイズがほぼ同等（109〜134件）でも
5月末は78分で完了していた処理が、6月以降は120分を超過するようになった。

---

## 影響

- **未着手投稿の完全な未スクレイピング**: 2026-06-21 は6件、2026-07-05 は30件が
  そのバッチで一切チェックされなかった（`updated_at` は更新されない）。
- **中断投稿の部分更新**: キャンセル直前に処理中だった投稿は、一部サービスだけ
  更新され残りは古いステータスのまま放置される
  （例: 2026-07-05 の `everything-everywhere-all-at-once-by-a24` は
  Netflix・Huluのみ更新され、それ以降のサービスは未チェックのまま終了）。
- **月次全件カバー保証の実質的な崩壊**: `BATCH_COUNT=4`（1ヶ月で1周）の
  設計は「バッジ内全件を1回のバッチ実行で処理しきれる」ことが前提だったが、
  この前提が成立しなくなっていた。未処理投稿は `updated_at` が古いままなので
  理論上は翌月の同バッチで優先されるが、同じ時間不足が続く限り
  **同じ投稿群が繰り返し取りこぼされ続けるリスク**があった。

---

## 解決

`BATCH_COUNT` を **4 → 8** に変更し、1バッチあたりの処理件数を約半減
（489件時点で ~122件/バッチ → ~61件/バッチ）することで、
1回の処理時間を120分のタイムアウト内に収める。

これに伴い、全投稿を1周する周期は **1ヶ月 → 2ヶ月** に変更された。

### バッチ番号算出方式も変更

旧方式（`(d.day - 1) // 7`、月の日付ベース）は月の日数差（28〜31日）により
特定バッチが月内に2回実行される歪みがあり、`BATCH_COUNT=8` に単純適用すると
1ヶ月では8バッチ分の日数を確保できず破綻する。

そのため、基準日（`_BATCH_EPOCH_MONDAY = 2024-01-01`、既知の月曜日）からの
**経過週数 % BATCH_COUNT** で算出する方式に変更した。月境界に依存せず、
週次実行（毎週月曜）を続ける限り必ず `BATCH_COUNT` 週で一巡する。

```python
def get_batch_for_date(d):
    weeks_elapsed = (d - _BATCH_EPOCH_MONDAY).days // 7
    return weeks_elapsed % BATCH_COUNT
```

### 変更箇所

| ファイル | 変更内容 |
|---|---|
| `weekly_patch.py` | `BATCH_COUNT = 4 → 8`、`get_batch_for_date()` を経過週数ベースに変更 |
| `main.py` | エンドポイントのdocstringを新方式に更新 |
| `utils/wordpress.py` | `get_all_posts_for_patch()` のdocstringを更新 |
| `.github/workflows/weekly-patch.yml` | `batch` 入力の説明を `0-7` に更新 |
| `tests/test_quota.py` | `get_batch_for_date()` / `get_post_badge()` のユニットテストを追加 |
| `docs/archive/weekly-patch-schedule.md` | スケジュール仕様・週次予算を全面更新 |
| `docs/archive/weekly-plan.md` | 「選択肢A」採用済みである旨を追記 |

### 未対応・今後のフォローアップ

- **実際にタイムアウトが解消されたかは未検証**。次回以降のGitHub Actions実行
  （batch 0〜7 の巡回）で `posts.total == posts.processed + posts.skipped + posts.errors`
  かつジョブが `success` で完了することを確認する必要がある。
- 投稿数がさらに増加した場合、再び120分を超過する可能性がある。その場合の
  追加選択肢（`timeout-minutes` 延長 / サービス切り替え待機の見直し /
  `BATCH_COUNT` のさらなる増加）は `docs/archive/weekly-plan.md` の
  「大規模対応」セクションを参照。
- サービス切り替え待機（10秒）がボトルネックの大部分を占めるため、
  投稿単位ではなくサービス単位でループする等の抜本的な高速化は未着手。
